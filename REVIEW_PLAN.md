# RAG 项目全面检查方案（Claude Code 分治策略）

## 为什么需要分治？

项目 22 个 Python 文件、14600+ 行代码。Claude Code 单次上下文约能精读 3000-5000 行。
直接让 Claude 一次性"检查所有代码"会导致：
- 前面的文件分析被压缩/遗忘
- 检查深度不够，只能做表面扫描
- 容易遗漏跨模块的逻辑问题

## 检查方案：5 轮分组 + 1 轮汇总

### 第 1 轮：核心检索引擎（约 3400 行）
**文件：** `rag_answer.py`(2177) + `search_utils.py`(1181)
**检查重点：**
- BM25 评分算法正确性
- 向量检索 + 混合排序逻辑
- 路由检测 → 章节定位 → 检索 → 排序 完整流程
- 边界条件：空查询、超长查询、无结果情况
- 并发安全性（全局状态、线程安全）

### 第 2 轮：配置 + NLP 处理（约 2900 行）
**文件：** `rag_runtime_config.py`(980) + `query_rewrite.py`(693) + `keyword_extractor.py`(609) + `synonym_store.py`(229) + `clarification_engine.py`(401)
**检查重点：**
- 路由关键词是否有遗漏/冲突/优先级问题
- 同义词展开是否正确
- 查询改写是否会破坏原意
- 多问题切分逻辑是否鲁棒
- 澄清引擎误触发风险

### 第 3 轮：API 服务 + 数据导入（约 3100 行）
**文件：** `api_server.py`(2288) + `import_knowledge.py`(661) + `media_router.py`(87) + `answer_formatter.py`(53)
**检查重点：**
- API 接口参数校验、错误处理
- XSS/注入等安全漏洞
- 知识库导入的 chunk 切分策略（与答案截断问题相关）
- 格式化输出是否丢失关键信息

### 第 4 轮：索引构建 + 辅助模块（约 1550 行）
**文件：** `build_faiss.py`(602) + `keyword_store.py`(422) + `relation_engine.py`(390) + `llm_client.py`(336) + `rag_logger.py`(203)
**检查重点：**
- FAISS 索引构建的向量维度一致性
- 关键词存储的增删改查正确性
- 关系引擎的实体关联逻辑
- LLM 调用的超时/重试/错误处理
- 日志是否暴露敏感信息

### 第 5 轮：测试代码审查（约 2700 行）
**文件：** `test_core.py`(1536) + `test_accuracy.py`(952) + `test_api.py`(401) + 三个 run_*.py
**检查重点：**
- 测试覆盖率是否有盲区
- 测试断言是否合理（不是太宽松）
- 回归测试用例是否充分

### 第 6 轮：跨模块汇总
- 收集前 5 轮发现的所有问题
- 分析跨模块数据流是否一致
- 检查 import 依赖关系是否有循环
- 汇总优先级排序、输出修复建议

## 使用方法

对 Claude Code 说：
```
按照 REVIEW_PLAN.md 的方案，执行第 N 轮检查，
精读每个文件，找出 bug、逻辑漏洞、性能问题和安全风险。
输出格式：文件名:行号 问题描述 严重程度(P0/P1/P2)
```

每轮完成后确认结果，再执行下一轮。
